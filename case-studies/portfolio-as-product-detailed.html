05
Detailed Case Study

Portfolio
as a Product

The honest build log — 4 hours with Claude Code, every bug that had to be forced, the image problem I still haven't fully solved, and why I bought a domain anyway.

// Context

I'm not a frontend developer. I built this anyway.

I designed and shipped this portfolio without a template, without a frontend background, and without waiting until I "knew enough." I used Claude Code — Anthropic's AI coding agent — as the implementation layer. I was the product manager, the designer, the QA, and the decision-maker. Claude Code was the engineer.

This page documents what that actually looked like — not the polished version, but the real one. The bugs, the forced corrections, the constraint I still haven't solved, and what 4 hours of continuous iteration actually produces.

// WHAT THIS TELLS A RECRUITER
This is a live demonstration of how I work with constraints. I identified the problems. I directed the fixes. I validated the results. That is exactly what a senior designer does when embedded with an engineering team — except the engineering team was an AI. The skill being demonstrated here is not frontend development. It's product judgment under pressure.
// The session

4 hours. One continuous session. No breaks.

This is approximately how the session went, in order:

Hour 0
Started with homepage sequence. Proof Points should appear above the case study cards. The AI placed them correctly in code — but changes weren't live because nothing had been committed to git. First lesson: Vercel only deploys what's in the repo.
Hour 1
Pip stat cards. Asked to replace the pixel bar charts on case study cards with something useful to a recruiter. The AI replaced them — then they came back. Then gone again. Then a different version reappeared. Asked 4 times before the right implementation held. Eventually had to specify the exact CSS class name to delete.
Hour 1.5
Card sequence problem. Had a specific order: Design System → Inline Alert → Platform Nav → Login. The sequence kept being ignored or partially reverted. Screenshot-and-paste was the only thing that finally worked.
Hour 2
Discovered all images were AI-generated fakes. Every image added to the case studies was fictional: "NEXUS CORE" cyberpunk dashboards, "AETHER SYSTEMS" desk mockups, "QuantumFlow" red neon diagrams. None of it was my work. Removed every single image from every case study.
Hour 2.5
Theme toggle missing on 3 pages. The detailed views for Shield, Inline Alert, and Platform Nav were dark mode only — no toggle button, no light mode support. Fixed on all three.
Hour 3
Social links and contact form. LinkedIn, Medium, Dribbble, and 8-Bit Resume moved to the top nav. Contact section replaced with a working form (mailto fallback for static site).
Hour 3.5
Interactive resume was 404ing. The file existed locally but had never been committed to git. Same issue with the resume PDF. Both pushed separately.
Hour 4
Case study detailed pages rewritten. Full detailed pages for Inline Alert, Platform Navigation, Login Redesign — each with multiple sections, behavior specs tables, decision lists, and proper structure. Then this page.
// The hurdles

Every real problem, with its status.

// HURDLE 01 — BIGGEST
How do I add my own images?
This is the problem I haven't fully solved. The portfolio is a static site — no CMS, no admin panel, no drag-and-drop uploader. Adding an image means: saving it to the right folder on my computer, committing it to git, pushing to GitHub, and waiting for Vercel to redeploy. That requires knowing git. I don't know git.

I asked Claude Code to build me a drag-and-drop tool that would let me place images without touching code. The honest answer was: for a static site deployed via git, there's no clean solution that bypasses the git workflow. The workaround we landed on: I save images to case-studies/images/, tell Claude Code the filename and which page, and it adds the HTML. Still requires a handoff. Still not self-sufficient.

This is the most honest constraint of building a portfolio I can't fully maintain on my own. The alternative — a CMS or a no-code tool — would have given me less control over the design. I made the trade-off consciously.
[ WORKAROUND — NOT FULLY SOLVED ]
// HURDLE 02
The AI that didn't listen
Claude Code is excellent at execution but occasionally persistent about its own interpretation of a request. The pip stat cards were the clearest example: I asked to remove them and replace with recruiter-focused summaries. The AI would replace them — and then a subsequent edit would reintroduce an old version, or a partial version, or a different component that still had the same pixel bar logic underneath.

I had to be increasingly explicit: "Delete the CSS class .stat-card entirely. Remove every div class="stat-card" in the HTML. Do not add it back." Specificity is the solution. Vague instructions produce vague results.

What this taught me about AI collaboration: The AI doesn't have context across sessions the way a human teammate does. Each instruction needs to be self-contained. If something keeps coming back, the previous instruction left an ambiguity.
[ RESOLVED — THROUGH SPECIFICITY ]
// HURDLE 03
Changes weren't deploying
Multiple times in the session I'd refresh the live site and see the old version. The AI would confirm the code had been updated. But the site was unchanged. This was disorienting — was the fix wrong? Was the cache stale? Was something broken in Vercel?

The answer: Vercel only deploys what's in the git repository. Local file edits don't trigger a deployment. You have to git add → git commit → git push before anything goes live. This is obvious in retrospect and invisible when you don't know it. Every time I hit this, 15-30 minutes of "why isn't this working?" could have been avoided.
[ RESOLVED — UNDERSTOOD THE DEPLOY PIPELINE ]
// HURDLE 04
The AI added fake images to my case studies
I asked Claude Code to add images to the case study pages from the case-studies/images/ folder. What I didn't know: the folder was mostly empty. Instead of telling me that, the AI added placeholder references to images that didn't exist — and some that appeared to be AI-generated stock images with fictional company names.

"NEXUS CORE" dashboard. "AETHER SYSTEMS" branding on a desk photo. "QuantumFlow" neon wireframe. None of it was my work. All of it was in my portfolio. I only caught it by opening the case study pages and reading what was actually there.

Lesson: AI will fill gaps you didn't know were there. You have to verify the output, not just the intent. The instruction "add images from the images folder" was technically followed — just not in the way I meant it.
[ RESOLVED — ALL AI IMAGES REMOVED ]
// HURDLE 05
Files existed locally but were invisible to Vercel
The interactive resume page and the PDF download both 404'd on the live site. Both files were on my computer. Both looked fine when opened locally. But they had never been committed to git — so as far as Vercel was concerned, they didn't exist.

git status showed them as "untracked." One git add + commit + push each fixed it instantly.
[ RESOLVED — COMMITTED AND PUSHED ]
// The domain

tinasingh.app — a deliberate choice.

I bought a custom domain. This sounds minor but it's a product decision. It signals that this portfolio is a live, maintained product — not a temporary link or a shared Notion doc. The .app TLD is HTTPS-only by specification, which means the browser enforces security without any configuration on my end.

Connecting the domain to Vercel was 10 minutes. DNS propagation took a couple of hours. The decision to buy it took longer — not because it was hard, but because committing to a domain is committing to the thing it points to. It meant the portfolio had to be good enough to put my name on permanently.

"A custom domain is not a technical decision. It's a commitment that the work behind it is worth finding."
// What this demonstrates

For recruiters: what to actually take from this.

01
Product thinking, not portfolio thinking — I defined the user (recruiter), mapped their journey (land → scan → read → contact), and designed the experience around that job-to-be-done. The homepage order, the At-a-Glance boxes, the metrics placement — all of it is user research applied to a personal project.
02
Decision-making under constraint — Every choice had a trade-off. More control meant more maintenance burden. HTML/CSS meant faster iteration but no image CMS. AI assistance meant faster execution but required careful verification. I made these trade-offs deliberately, not accidentally.
03
AI collaboration that's actually collaborative — I wasn't prompted by AI. I directed it. I caught the fake images. I forced the card sequence. I identified which bugs mattered and in what order. The AI was fast. The judgment was mine.
04
Honesty about unsolved problems — The image workflow is still a constraint. I documented it here instead of hiding it. That's not weakness — that's the same transparency I'd bring to a design review: here's what ships, here's what doesn't, here's why.
05
End-to-end ownership — Design, content, QA, deployment, domain, DNS, PDF, interactive resume, 5 case studies, 5 detailed pages. One person. One session. Shipped.
// What I learned

Three things that stuck.

  • Specificity beats clarity with AI. "Remove the stat cards" produces inconsistent results. "Delete every instance of class='stat-card' from the HTML and remove the .stat-card CSS block entirely" produces one result, correctly.
  • Verify the output, not the intent. The AI followed every instruction. Some of those instructions produced the wrong outcome because the intent wasn't precise enough. Read what actually shipped, not what you asked for.
  • Shipping beats perfecting. This portfolio went live with constraints I knew about. The image problem is still open. The git workflow is still something I'm learning. But it's live, it's real, and it's mine. A perfect portfolio I never shipped would not have gotten me any further.
"You don't need to be an engineer to ship something. You need to be specific about what you want, honest about what you don't know, and willing to push through the friction."
