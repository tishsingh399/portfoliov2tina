The Problem

Alert fatigue in high-stakes environments.

Enterprise security products deal in urgency. Every status needs attention. Every error matters. But when the UI treats every message with the same weight — banners stacking on banners, red everywhere — users stop reading.

Across BeyondTrust's product suite, alert patterns had evolved organically. Different teams used different components, different severities, and different placements. The result:

  • Users ignoring critical alerts because they were buried in noise
  • No consistent hierarchy — same visual weight for "session expired" and "privilege escalation detected"
  • Banner overload — some screens stacked 3-4 persistent banners simultaneously
  • Accessibility failures — color-only severity indicators with no semantic markup
  • No clear pattern for contextual vs. global alerts
"The problem wasn't missing alerts. It was that every alert screamed at the same volume — and users learned to stop listening."
The Solution

A three-tier alert hierarchy.

I designed a structured alert system based on one principle: proximity to context determines the pattern. Global issues get banners. Contextual issues get inline alerts. Transient confirmations get toasts.

▬
Banner
System-wide, persistent. Sits at the top of the viewport. For states that affect the entire session.
Global · Persistent · Max 1
▸
Inline Alert
Contextual, positioned near the relevant content. For field-level or section-level feedback.
Contextual · Persistent · Unlimited
◦
Toast
Transient, auto-dismissing. For confirmations and non-blocking status updates.
Transient · 5s Auto-dismiss · Queue

Each tier has distinct visual language, ARIA roles, and interaction rules. They never compete — they complement.

Key Decisions

Three principles that shaped the system.

01
Proximity determines the pattern

If the issue is global, it's a banner. If it relates to a specific field or section, it's inline. If it's a confirmation of an action, it's a toast. The placement logic is deterministic — teams don't decide which pattern to use, the context decides for them.

02
Severity is not just color

Every alert tier uses icon + color + text to communicate severity. Color alone fails accessibility. I defined four severity levels (info, success, warning, critical) with distinct iconography, ARIA labels, and contrast-checked color tokens that work in both light and dark modes.

03
Constrain the noise by design

Banners have a hard cap: one at a time. If a second system-level issue arises, it queues. Toasts auto-dismiss after 5 seconds with a hover-to-pause rule. These constraints aren't limitations — they're the design doing its job.

Process

Audit, define, test, ship.

01
Alert audit — Cataloged every alert pattern across 6 products. Found 47 unique alert implementations with no shared API or visual language.
02
Hierarchy definition — Defined the three-tier model with clear usage rules. Documented when each tier applies and when it doesn't.
03
Accessibility pass — Ensured all severity levels meet WCAG AA. Used AI-assisted scanning to identify contrast failures across existing implementations.
04
Component build — Designed Banner, InlineAlert, and Toast components in Figma with full token integration. Spec'd interaction states, keyboard nav, and screen reader behavior.
05
Migration — Worked with product teams to migrate legacy alerts to the new hierarchy. Prioritized pages with the highest alert density first.
Outcome

What changed.

Before
47 different alert implementations. No hierarchy. Users ignoring critical security notifications.
After
3-tier system with deterministic placement rules. 60% reduction in alert noise. Users engaging with critical alerts again.
Before
Color-only severity. Multiple WCAG failures. No ARIA support for alert announcements.
After
Full WCAG AA compliance. Icon + color + text severity. Live-region ARIA announcements for all tiers.
What I Learned
"Solving alert fatigue isn't about better alerts. It's about fewer of them in the right places — and the discipline to enforce that constraint."

In enterprise security, every stakeholder believes their alert is the most important. The design challenge wasn't visual — it was political. Building a system that constrains noise requires convincing product teams that less is more. The hierarchy gave us a shared vocabulary to have those conversations.
