Inline Alert

Security engineers were dismissing critical errors because every alert looked the same. That's not an alert problem.

// AT A GLANCE
ROLE UI Designer · Component & Pattern Design · Design Systems
SCOPE 6 product areas, security operations platform — enterprise B2B
PROBLEM A single visual style covered all severity levels. Security engineers were dismissing critical alerts because everything looked the same — alert fatigue at the UI layer
  • Defined a 4-tier severity system (Critical / Warning / Info / Success) with distinct iconography, color tokens, and dismissal behavior — reducing misread rate in testing
  • Adopted platform-wide across all 6 product suites with zero rework — the component was system-ready from day one
  • WCAG AA contrast baked directly into semantic tokens — compliance became automatic, not a post-launch audit

UI Designer, Platform Design

I audited message types across six products and found that 40% of what teams were calling a "banner" was actually a form-level error. It just didn't live where the form was. The component wasn't the problem. The category was.

The Problem

Banner blindness.

In complex workflows, users were overlooking critical errors because the system used global banners for everything, from system outages to "save successful" messages. When I audited message types across six product areas, I found that 40% of what teams called "banners" were actually contextual errors that belonged near the form field causing the problem, not at the top of the screen.

The core insight: proximity. An error message is only useful if it's visible when the user is trying to fix it. A banner at the top of a page doesn't help someone filling out a form at the bottom.

01
No hierarchy: Banners, inline messages, and toasts were used interchangeably, with no shared taxonomy across 6 products.
02
Alert fatigue: Stakeholders wanted all alerts sticky and permanent. I pushed back: if every announcement is permanent, users tune them out, including actual security warnings.
03
Missing middle layer: Banners (global) and toasts (transient) existed. Nothing sat contextually, inline, next to the source of an error.

// Banners (global) · Inline Alerts (contextual) · Toasts (transient) — the three-tier taxonomy

The Solution

A clear taxonomy. Negotiated across 7 teams.

Dec 2024 – May 2025. I led the definition, design, and rollout of the feedback taxonomy. Not just the component in Figma, but the usage contracts with 7+ product teams that determined whether the pattern would actually stick or get quietly ignored.

01
Taxonomy first, component second

I shifted the question from "What does it look like?" to "What is it doing?" Four severity levels: Critical (blocks work, permanent), Warning (proceed with caution), Info (contextual, no action needed), Success (user-initiated confirmation). Only errors that stop work are permanent. Everything else is dismissible.

02
Token-first architecture

Token structure: color.feedback.*, spacing.xs, radius.sm. WCAG 2.1 AA verified via Stark. Chromatic automated regression builds prevented style leaks across legacy code. Critical for engineering trust.

03
Migration guides as the adoption lever

Audited existing flows (including the Login Redesign) to prove 40% of "banners" were actually contextual errors. Used that data to persuade teams. Wrote migration guides mapping hardcoded values to semantic tokens. Made adoption the path of least resistance.

Reflection
"Global banner misuse dropped 70% after rollout. Not because we removed banners, but because teams finally had something more specific to reach for."

A lot of the work was persuasion, not design. Showing teams with audit data that what they'd been calling banners were actually form errors, then making the switch low-effort with migration guides. The component shipped. The harder thing was getting seven product teams to agree on what each type of message was for.
