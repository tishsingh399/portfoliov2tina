Navigation isn't a component.
It's an argument about product structure.

The Problem

Three product suites. No shared navigation logic.

BeyondTrust's platform spans multiple product suites — each with its own navigation patterns, hierarchy rules, and tenant-switching behavior. As the platform grew, the navigation diverged:

  • Each product suite defined its own sidebar hierarchy independently
  • Multi-tenant switching worked differently across products — some used dropdowns, others used separate login flows
  • No consistent behavior for what happens when a user switches context (tenant, product, or scope)
  • Breadcrumb patterns contradicted sidebar hierarchy in multiple products
  • Engineering teams had no shared specification — each rebuilt navigation from assumptions
"We didn't have a navigation problem. We had a product-structure problem that hadn't been made explicit."
My Approach

Define behavior before visuals.

Most teams start navigation redesigns with wireframes. I started with a behavior specification. Before deciding what the sidebar looks like, I needed to answer:

  • What are the navigation levels and how do they nest?
  • What happens when a user switches tenants mid-workflow?
  • Which state persists across context switches and which resets?
  • How does the hierarchy work for single-product vs. multi-product users?

This produced a behavioral spec — a document that defined navigation rules without any visual design. Engineering reviewed it first. Visuals came after behavior was agreed upon.

Navigation Architecture

Four levels. Clear hierarchy.

L1 · PLATFORM
Product suite selection. Top-level context — which product surface is the user in? Persistent across the session. platform.context
L2 · TENANT
Tenant/organization switching. Which customer environment? Switching resets L3 and L4 state. tenant.active
L3 · MODULE
Primary sidebar navigation. Feature modules within the product. Collapse/expand behavior with deep-link support. module.current
L4 · VIEW
Sub-navigation and tabs. Views within a module — list/detail, settings panels, nested workflows. view.active

The key rule: switching a higher level resets all lower levels. This prevents orphaned state — a user never lands in a view that doesn't belong to their current tenant or module.

Key Decisions

Three decisions that shaped the architecture.

01
Write the spec before the wireframe

I wrote a behavioral specification first — defining state management, context-switching rules, and hierarchy constraints. Engineering reviewed it and gave feedback before I opened Figma. This meant the visual design was implementing an agreed-upon contract, not proposing one.

02
Tenant switching resets downstream state

When a user switches tenants, module and view state resets to defaults. This prevents data leakage (showing one tenant's data in another's context) and eliminates an entire class of edge cases. Some product teams pushed back — but the alternative was unpredictable state across every product.

03
One navigation model, flexible rendering

The behavioral model is shared across all products. But visual rendering can differ — one product might render L3 as a sidebar, another as a tab bar. The hierarchy and state rules stay the same. This gave product teams flexibility without fragmenting the architecture.

Process

Behavior → spec → design → ship.

01
Cross-product audit — Mapped navigation patterns across all 3 product suites. Documented hierarchy, state behavior, and tenant-switching flows.
02
Behavioral specification — Wrote a detailed spec defining the 4-level hierarchy, state management rules, and multi-tenant switching logic. No visuals in this document.
03
Engineering review — Shared the spec with engineering leads across all product teams. Three rounds of feedback refined edge cases around deep linking and session persistence.
04
Visual design — Designed the navigation components in Figma based on the agreed spec. Built responsive variants for sidebar, collapsed, and mobile states.
05
Phased rollout — Rolled out to the primary product suite first, validated behavior with real user sessions, then expanded to remaining suites.
Outcome

What changed.

Before
3 product suites with independent navigation. No shared hierarchy or state model. Breadcrumbs contradicting sidebars.
After
Unified 4-level navigation architecture. Deterministic state management. Shared behavioral contract across all products.
Before
Tenant switching was inconsistent — some products used dropdowns, others required re-login. State leaked across contexts.
After
Consistent tenant switching with predictable state reset. No data leakage. Users always know where they are.
What I Learned
"Navigation is the most political component in enterprise software. Everyone has an opinion about the sidebar — but almost no one has defined the rules behind it."

The biggest insight: visual design is the last 20% of navigation work. The first 80% is defining behavior — state management, hierarchy rules, and multi-tenant logic. Starting with wireframes would have locked us into visual debates while the actual problems (state leakage, inconsistent hierarchy) went unsolved.
