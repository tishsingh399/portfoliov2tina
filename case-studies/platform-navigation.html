Platform Navigation

Four acquisitions, four navigation systems. Users felt like they were changing apps every time they switched tools.

// AT A GLANCE
ROLE UX Architect · Information Architecture · Interaction Spec
SCOPE 6 acquired products, 4 post-acquisition codebases, enterprise security platform
PROBLEM 4 acquisitions brought 4 navigation systems. Users switching products felt like changing apps — 40% of menu links were redundant, active states were contradictory, deep links broke between suites
  • 40% navigation redundancy eliminated — restructured into a Hub & Spoke IA model with a global bar + contextual side rail
  • 1,000+ deep-link routes unified through a single JSON payload — no per-product rewrites, 100% adoption in 4 months
  • 22% faster cross-product workflows — Behavioral matrix (active states, tenant switching, breadcrumb logic) delivered as engineering handoff, not just Figma

Nobody had written down what "pinned" meant. Or what should happen to a pinned item after a tenant switch. Or what the active state for a nav icon should be called. Six products. Six different answers to the same questions. That's what I was actually solving.

The Problem

Navigation as a bottleneck.

By the time the platform had six products, navigation was slowing down work instead of enabling it. Behavior changed between products. Users lost their place. Developers maintained separate codebases for the same UI because there was no shared version to point to. Each new product integration meant rebuilding navigation from scratch.

The problem was architectural, not visual. No shared logic for states like pin/unpin, no persistence model, no defined behavior for hierarchy and tenant switching. What looked like a styling problem was actually undefined behavior.

// Before: fragmented, annotated with issues. After: unified hub-and-spoke

My Approach

State maps before mockups.

01
Replaced static mocks with full lifecycle prototypes

Documented every state: Hover, Focus, Active, Pinned, Expanded, Collapsed, in both light and dark mode. This gave stakeholders behavior to react to, not just pixels. The UXA team used the state map to build a functional proof of concept immediately after handoff.

02
Pin/unpin: three prototypes, one decision

To define the pinned state, I built three distinct prototypes: a solid pin for clarity, an outline pin with an accent ring for dark mode visibility, and a dynamic motion-based indicator. Each was fully interactive and linked to design tokens. The final model uses action-based labels, instant persistent feedback, and 160ms fade and scale transitions aligned to SDS tokens.

03
Token-driven theming throughout

Colors, typography, spacing, and elevation mapped to semantic tokens, including icon.action.active for built-in accessibility and theming. The nav is consistent in light and dark mode and aligned to the design system without manual overrides.

The Impact
→
Engineering built a working POC immediately from the state map. No back-and-forth on "what does hover do?" or "what happens to pinned items after tenant switch?"
→
Tenant switcher simplified from a search box to a dropdown in the top global bar. Users got a consistent "where am I?" anchor across all products.
→
Defined behavior became the system standard for future platform components. The logic introduced here prevented future navigation debt.
"I built three fully interactive pin/unpin prototypes before we settled on one. The team needed to react to real behavior, not a description of it. The state map went to engineering the next day."
