Detailed Case Study

Login Redesign

A System Fix Disguised as a Small Update — how a "simple visual refresh" became a foundational rebuild of the authentication entry point.

// How it started

The request sounded small. It wasn't.

The initial ask was a refresh: update the logo, fix the colors, make it match the new brand. But when I pulled the page apart, it was clear this was a structural problem, not a visual one.

The login page was written as standalone code — it lived completely outside the Shield Design System. It used hardcoded hex values, imported its own fonts separately, and had no relationship to the token system or component library the rest of the product ran on. Every visual decision was a one-off. Nothing was governed.

"The login page is the front door. For a PAM security product, if the front door looks off, security professionals immediately question whether it's a phishing attempt."
// What I found

Three layers of broken.

I ran a full audit before touching any design. What I found fell into three categories:

01
Brand disconnect — The login page used an old background photo, a previous iteration of the logo, and typography that predated our current font system. It looked like a different product from a different company.
02
Error state gap — There was one error message: "Login Failed." No specificity for expired SSO sessions, locked accounts, invalid MFA codes, or tenant mismatches. Every failure looked identical. Support tickets piled up because users couldn't self-serve.
03
Accessibility failures — Color contrast ratios fell below WCAG AA on multiple states. Focus indicators were missing. The tab order was broken. Federal agency customers — a major segment — had specific compliance requirements we were actively failing.
// The scope shift

Reframing the project.

I went back to the PM and engineering lead with the audit findings. This wasn't a visual refresh — it was a system integration project with an accessibility mandate attached. We needed to:

  • Bring the login page into the Shield Design System completely — no more standalone code
  • Map every IAM error state to a specific, actionable UI response
  • Implement a progressive disclosure flow to reduce cognitive load and support SSO users
  • Achieve WCAG AA compliance across all interactive states

The scope change was approved. The "small update" became a four-week project with IAM engineering involvement.

// Key design decisions

Three decisions that defined the solution.

01
Progressive disclosure over a single form

The old design showed every field simultaneously — email, password, SSO toggle — regardless of what authentication method applied to that user. I split it into two steps: Step 1 collects the email, Step 2 shows only what's relevant (password field, SSO redirect, or MFA prompt). SSO users go straight to their identity provider without ever seeing a password field. The form got shorter. The path got faster.

02
Error taxonomy mapped to IAM logic

I sat with the IAM engineering team for two sessions and walked through every possible failure path the backend could return. We categorized them into 15+ specific states and matched each to a UI response — not just a message, but an action. Expired sessions got a re-authenticate link. Locked accounts got admin contact info. Tenant mismatches got a tenant selector. "Login Failed" was retired.

03
Design System integration as a requirement, not an upgrade

I enforced a rule: no hardcoded values. Every color came from a semantic token. Every spacing unit came from the spacing scale. Every component — inputs, buttons, error states — came from Shield. This meant the login page would automatically inherit any future system updates, dark mode support, and accessibility improvements without a dedicated engineering pass.

// Accessibility implementation

Compliance wasn't a checkbox — it was a constraint.

Federal agency users have mandated WCAG AA requirements. Failing that meant disqualification from procurement. I treated accessibility as a hard requirement from the start, not a QA pass at the end.

■
Color contrast — All text/background combinations verified to meet 4.5:1 minimum ratio. Error states tested in both light and dark mode.
■
Focus management — On step transition (email → auth method), focus moves programmatically to the first new field. No user has to hunt for where they are.
■
Screen reader labels — All inputs, error messages, and status states have explicit ARIA labels. Error messages are announced on trigger, not just visually shown.
■
Keyboard navigation — Full tab order validated. The form is 100% operable without a mouse, including the SSO redirect and tenant selector.
// Additional scope: supporting flows

The login experience doesn't start at the login page.

Once the core page was solved, I extended the same rigor to three adjacent surfaces that were part of the same authentication journey:

  • Invite emails — Refreshed branding, updated logo and product name references, updated deep-link format for new URL structure.
  • Tenant selector (Workforce Passwords) — Added a portal dropdown so admins could distinguish which environment they were accessing — a real confusion point in multi-tenant setups.
  • MFA confirmation screen — Added a countdown timer for TOTP codes, replacing the static instruction text that gave no signal on expiry.
// Outcome

What changed and what I learned.

The redesign shipped. The login page now runs fully on Shield Design System tokens — no exceptions. Future system updates (dark mode, rebranding, accessibility patches) propagate automatically.

  • SSO-related support tickets dropped — users could now self-identify their error and take action without contacting support
  • WCAG AA compliance achieved, removing a procurement blocker for federal accounts
  • Engineering handoff was cleaner than any previous login work — everything mapped to the design system, no custom CSS to negotiate
"The biggest lesson: scope creep is sometimes scope correction. What felt like a bigger ask was actually the right-sized fix."

If I'd shipped just the visual refresh, we'd have updated the logo and still had 15 broken error states. The audit forced the right conversation. That's the job — not just making it look better, but making it actually work.
