The Problem

Design debt at enterprise scale.

BeyondTrust's product suite — 8 enterprise security products across Angular and React stacks — had no shared design language. Each product team maintained its own component files, color values, and interaction patterns.

The consequences were measurable:

  • Inconsistent UI across every product surface
  • No Figma-to-code parity — designers and engineers worked from different sources of truth
  • Duplicated effort across teams building the same components differently
  • No token architecture — color, spacing, and type values were hardcoded everywhere
  • Accessibility gaps compounding across products with no systematic remediation path
"The problem wasn't that we had no design system. It's that we had eight of them — all unintentional."
My Role

Systems architect for design infrastructure.

I was brought in to build the design system from zero. Not to adopt an existing framework, not to reskin what existed — to define the foundation that would unify the entire product suite.

I led the system end-to-end: defining the token architecture, designing and documenting 340+ components, establishing governance processes, and working directly with engineering teams across both Angular and React stacks to ensure parity.

Token Architecture

The FORK model.

The core design decision was the token architecture. I developed the FORK token model — a system where primitive tokens never chain through semantic tokens. This prevents cascading breakage and makes the system predictable at scale.

Layer Role Example
Primitive Raw values — the atomic palette color.blue.500
Semantic Intent-mapped — what values mean color.action.primary
Component Scoped to component context button.bg.default

The key insight: semantic tokens reference primitives directly, never other semantics. Component tokens reference semantics. This creates a clean dependency graph that doesn't tangle when you need to change a color across 8 products.

Key Decisions

Four decisions that defined the system.

01
Start with tokens, not components

Most teams start by building a button. I started by defining the token architecture. Without a shared value system, components are just more inconsistent UI wearing a uniform. Tokens first meant every component was born consistent.

02
Build for two stacks simultaneously

Angular and React have fundamentally different component models. I designed the system to be framework-agnostic at the token and specification layer, with framework-specific implementations that share the same visual and behavioral contract.

03
Embed governance from day one

A system without governance is a suggestion. I established contribution guidelines, review processes, and documentation standards before the first component shipped. This meant adoption was structured, not chaotic.

04
Treat Figma as code

I used Tokens Studio to sync design tokens between Figma and code repositories. This meant designers and engineers were always working from the same source. No more "it looks different in the spec" conversations.

Process

From audit to adoption.

01
Audit — Cataloged every component, pattern, and color value across all 8 products. Identified 200+ unique button variants alone.
02
Token foundation — Defined the FORK token architecture. Built the primitive palette, semantic mappings, and component token structure in Tokens Studio.
03
Core components — Designed the first 40 foundational components (buttons, inputs, alerts, modals, tables) with full specification documentation.
04
Engineering handoff — Worked with Angular and React teams to implement components with token parity. Validated output in Storybook.
05
Migration + scale — Migrated existing product UIs to the new system. Expanded to 340+ components with governance and contribution workflows.
Outcome

What changed.

Before
8 products with independent, inconsistent UI. No shared tokens. No Figma-code parity.
After
Unified design system with 340+ components, FORK token architecture, and full Figma-code parity across Angular + React.
Before
Design debt compounding with every new feature. Engineers guessing at specs.
After
Governed system with documentation, contribution guidelines, and predictable component behavior across stacks.
What I Learned
"A design system isn't a component library. It's an argument about how your products should behave — made tangible in tokens, components, and governance."

The hardest part wasn't building the components. It was building consensus. Getting 8 product teams to agree on a shared language required more persuasion than pixels. The technical architecture was the easy part — the organizational architecture was the real design challenge.
